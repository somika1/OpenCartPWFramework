Q. what is Schema validation in API? why is it required?
Ans. Schema validation is one of the most important concepts in API testing, especially in automation. 
It verifies that an API response follows the expected structure and data types, regardless of the actual values.
It is like a blueprint or contract that defines:-
1. Which fields should be present
2. Which fields are required
3. Data types of each field
4. Allowed values (optional)
5. Nested object structure
6. Array structure
7. Constraints (length, min/max, regex, etc.)

OAuth 2.0 is an authorization framework that allows a client application to access protected resources 
on behalf of a user without sharing the user's credentials. 
Instead of passing the username and password to every API, the authorization server authenticates the user and 
issues an access token. The client includes this token in the Authorization: Bearer <token> header when calling 
protected APIs. If the access token expires, a refresh token can be used to obtain a new one without requiring 
the user to log in again. A common real-world example is "Login with Google," where Google authenticates the 
user and issues a token that allows another application, such as Spotify, to access the user's profile information
with the user's consent.

Q: How do you test an OAuth-protected API?
1. Request an access token from the authorization server (often using the appropriate grant type, client secret and client id).
2. Extract the access_token from the response.
3. Include it in the request header:
4. Authorization: Bearer <access_token>
5. Validate the API response.
6. Also test negative scenarios such as an expired token, an invalid token, or missing required permissions (scopes).

grant type, client secret and client id is not sent in form of json body but called as form payload

*** Interception ******
--------------------------
Interception in Playwright means capturing network requests before they reach the server. Using page.route(), 
we can inspect, modify, mock, continue, or abort HTTP requests and responses. 
It is commonly used to test UI behavior without depending on backend availability, 
simulate different server responses (such as 404 or 500 errors), modify headers or request payloads, 
block unnecessary resources, and make automated tests more reliable and faster.
